# Figure Selection Heuristics

Choose figures by paper function, not by habit.

## General Rule

If a paragraph makes a reader ask "what does this system or result actually look like?", the paper probably needs a figure there.

## Minimal Starter Pack For A Method Paper

If a paper has few or no figures, propose a starter pack of roughly five figures:

1. motivation or failure example
2. system overview
3. internal mechanism or contract diagram
4. runtime or execution loop
5. evaluation comparison figure

Add more only when they carry distinct explanatory work.

## Match Section To Figure Type

- Introduction or Motivation:
  use a motivating scenario, failure taxonomy, or before-versus-after view.
- Approach overview:
  use a wide system architecture or pipeline figure.
- Internal method section:
  use a bus, state machine, algorithm flow, graph structure, or module interaction diagram.
- Runtime section:
  use a protocol or execution loop spanning components.
- Evaluation section:
  use bar charts, line charts, ablation summaries, subject matrices, or benchmark dashboards.
- Discussion section:
  add no new figure unless it clarifies a threat, limitation, or deployment story better than text.

## When To Keep An Existing Figure

Keep it if all of the following are true:

- it already supports a clear paper claim
- terminology is current
- file naming and TeX references are stable
- visual style is consistent with the rest of the paper
- the figure survives reduction without becoming muddy

## When To Refine Instead Of Redraw

Refine if the structure is good but the figure suffers from one or two local issues:

- caption drift
- inconsistent palette
- label clutter
- weak legend placement
- minor layout imbalance

## When To Regenerate From Scratch

Regenerate if any of the following are true:

- the figure no longer matches the implementation or paper story
- terminology changed across the codebase or docs
- the figure is decorative but not explanatory
- it relies on inconsistent visual grammar relative to the rest of the paper
- it is easier to redraw cleanly than to patch repeatedly

## Prefer TikZ When

- the figure is schematic or diagrammatic
- the figure should inherit paper typography
- the figure may need many small revisions
- the paper benefits from consistent semantic colors and line styles

## Prefer External Vector Assets When

- the geometry is too complex for maintainable TikZ
- the figure is generated by another plotting pipeline
- the team already has a reliable vector export source

## Prefer Raster Only When

- the figure is a screenshot or visual evidence from the system itself
- a vector recreation would lose essential realism

## Result Figure Policy

- Use real data when it exists.
- Use illustrative placeholders only when the paper text already frames results as pending.
- Never present placeholder bars or lines as if they were measured outcomes.

## Naming Guidance

- Use stable semantic names.
- Prefer `fig_<topic>.tex` for TikZ source.
- Use one source file per figure unless a shared plotting pipeline already exists.

## Final Sanity Check

Before drawing a new figure, answer these questions internally:

- What exact sentence or claim in the paper does this figure support?
- Why is a figure better than a table or paragraph here?
- What evidence from the repo anchors the content?
- Is the figure conceptual, measured, or illustrative?
